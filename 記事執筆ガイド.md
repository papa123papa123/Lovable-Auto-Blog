# 記事執筆ガイド（LLMプロンプト用）

SEO記事を執筆する際の基本方針、心理的アプローチ、執筆ルール、プロンプトテンプレートを統合したガイドです。

---

## 1. 基本方針：読者中心の執筆アプローチ

### 1.1 核となる考え方

SEO記事は、読者が検索した瞬間に「頭の中にある問い」に答えるコンテンツです。  
**相手の頭の中を想像できているか**で勝負が決まります。

### 1.2 「ノー→イエス」3ステップ

**ステップ1：自分目線の説明を捨てる**
- ×「◯◯とは〜です。定義は〜。」
- ○ 冒頭で読者の痛みを言語化し、「結論→理由→手順」の順に並べる

**ステップ2：検索者の頭の中を文章化する**
- 読者の脳内メモ：不安（失敗したくない）× 欲望（得したい/ラクしたい）× 状況（時間/お金/スキル/期限）
- 見出し＝読者の頭の中の順番

**ステップ3：読者メリットと一致する答えを作る**
- 行動を正当化する一文で締める
- ×「有料プランにしましょう」
- ○「『商用利用の不安を消す』『サポートで詰まらない』ために、ここだけは有料の方が結果的に安い」

## 2. 心理的アプローチ

### 2.1 「イエス」に変える7つの切り口

1. **相手の好きなこと**（導入1-3行目、H2の言い回し）
   - 「結局どれが一番ラクで安い？」

2. **嫌いなこと回避**（不安が強い読者に効く）
   - 「戻せない事故が起きた瞬間、全部が終わります。だから"最小の保険"だけ先に」

3. **選択の自由**（A/Bの2択で迷いを解消）
   - 「あなたはどっち？」→「当てはまる方を選べばOK」

4. **認められたい欲**（読者を肯定する）
   - 「ここ、誰でも詰まります。むしろここまで来てる時点で十分すごい」

5. **あなた限定**（導入2-3行目、まとめ直前のCTA）
   - 「初心者のあなたにだけ、先に言います」

6. **チームワーク化**（心理的ハードルを下げる）
   - 「一緒にここだけ設定しましょう。ここさえ通れば、あとは流れ作業です」

7. **感謝**（読者の行動に対して感謝）
   - 「ここまで読んでくれてありがとう。あなたは"調べる力"がある」

### 2.2 「強いコトバ」をつくる8つの技術

1. **サプライズ法**：言い切りや意外性で注意を奪う
   - 「広告が出ない原因、9割は"ここ"です」

2. **ギャップ法**：逆を置いて後半でひっくり返す
   - 「難しそうに見えるけど、やることは3つだけ」

3. **赤裸裸法**：体感を入れて映像化する
   - 「ログを見た瞬間、背中が冷えた——原因は"設定"じゃなく"配信"だった」

4. **リピート法**：同じキーワードを繰り返す
   - 「迷うなら、まず"切り分け"。切り分けができれば、勝ちです」

5. **クライマックス法**：導入で結論を先出し
   - 「結論：広告が出ないのは"設定"じゃないケースが多い。確認順はこの3つ」

6. **ナンバー法**：数字で言い切る
   - 「失敗が減るチェックは"7つ"」

7. **合体法**：既存概念に別軸を足す
   - 「設定沼」×「安全復旧」→「安全復旧ランブック」

8. **頂上法**：言える範囲で頂上ワードを入れる
   - 「初心者が最短で失敗しない、唯一の順番」

## 3. 執筆ルール

### 3.1 文字数ルール

- **記事全体**：10,000〜20,000字（厳守）
- **各H3**：800字以上（必須）
- **H2導入文**：200-300字

### 3.2 文章構成ルール

**スマホ読者意識**
- 1文：30〜40字以内（最大40字）
- 1段落：2〜3文（50〜80字程度）
- 適度な改行でリズムを生む
- スキャン読みで要点が掴める構成

**結論ファースト構成（H3）**
- 1文目：結論を書く（見出しの疑問に対する答え）
- 2文目：理由または条件
- 3文目以降：表・吹き出し・リストを配置
- 途中から読んでも理解できる独立完結型

### 3.3 装飾要素（必須）

**吹き出し会話**（各H3に1-2回）
```html
<div class="bubble-left"><span class="bubble-icon">Q</span><div class="bubble-text">短い質問文</div></div>
<div class="bubble-right"><span class="bubble-icon">A</span><div class="bubble-text">具体的で短い回答</div></div>
```

**マーカー強調と太字**（多用すること）
```html
<mark style="background: linear-gradient(transparent 50%, rgba(254,240,138,0.7) 50%);"><strong>重要ポイント</strong></mark>
<mark style="background: transparent; border-bottom: 3px solid rgba(96,165,250,0.5);"><strong>数字・期限</strong></mark>
<mark style="background: linear-gradient(transparent 50%, rgba(251,207,232,0.7) 50%);"><strong>注意事項</strong></mark>
<mark style="background: linear-gradient(transparent 50%, rgba(187,247,208,0.7) 50%);"><strong>メリット</strong></mark>
```

**情報テーブル**（各H3に1個、3列以内）
```html
<table class="info-table">
  <tr><th>項目</th><th>内容A</th><th>内容B</th></tr>
  <tr><td>項目名</td><td>説明</td><td>説明</td></tr>
</table>
```

**チェックリスト**
```html
<ul class="check-list">
  <li>具体的なチェック項目</li>
</ul>
```

**ボックス**
```html
<div class="info-box">📌 まとめ・ポイント</div>
<div class="warning-box">⚠️ 注意事項</div>
<div class="ok-box">✅ 安心ポイント</div>
```

**外部リンク**（記事全体で6個以上必須）
- EETを満たす権威性サイトの個別ページにリンク（.go.jp、.or.jp、.ac.jp、大手企業公式サイトなど）
- craw4ai/firecrawlで実際に生きたURLを検索して使用
- 大手のトップページではなく、具体的な情報がある個別ページにリンク
- ハイパーリンク形式：`<a href="URL" target="_blank" rel="noopener noreferrer">リンクテキスト</a>`
- リンクは記事内に自然にちりばめる（各H3に1-2個、H2導入文にも1個）
- リンクテキストは自然な文脈で、何へのリンクか明確に（例：「厚生労働省のガイドラインによると」）
- リンク切れ確認必須（検索で確認済みURLのみ使用）

### 3.4 禁止事項

- 「〜について解説します」「〜を見ていきましょう」等の前置き
- 「まず」「次に」「ここでは」等の接続詞での水増し
- 同じ内容を言い換えて繰り返す
- 結論を最後まで引っ張る構成
- 「〜かもしれません」「〜の可能性があります」で逃げる
- ポエム的寄り添い（「不安ですよね」「心配になりますよね」）
- 前振り・次章誘導（「詳しくは後述」「次の章で解説」）
- 検索結果にない数字を創作する（ハルシネーション）
- **太字マークダウン記法**（必ずHTMLの`<strong>`タグで出力）
- *イタリックマークダウン記法*（必ずHTMLの`<em>`タグで出力）
- #見出しマークダウン記法（見出しは出力しない、またはh2/h3タグのみ）
- 長い段落（4文以上の連続）
- 1文60文字以上
- 吹き出しアイコンに「Q」「A」以外を使う

### 3.5 曖昧語変換ルール

| × 禁止 | ○ 推奨 |
|--------|--------|
| 場合による | ◯◯なら△△、□□なら×× |
| 一般的に | メーカー公式では / 専門家によると |
| 〜と言われている | ◯◯（出典）によると |
| 個人差がある | ◯◯の人は△△、□□の人は×× |
| しばらく | ◯分 / ◯時間（検索で取得） |
| 高額になる | ◯〜△万円（検索で取得） |

## 4. プロンプトテンプレート

### 4.1 システムプロンプト構造

```
あなたはAI要約では満足できない読者の「自分のケースはどうなの？」に答える編集者です。
一般論ではなく「境界条件」「判定基準」「最悪ケース」を具体的に示します。

【重要：現在の日付】
今日は{currentDate}です。「最新」「今年」などの表現は{currentYear}年を基準にしてください。

【セクションの役割】
{H2-1またはH2-2の役割説明}

【最重要：断言ルール - Groundingで裏を取る】
★必ずGoogle検索で最新情報を取得し、その結果に基づいて断言する★

1. 数字で断言する
   - 検索で「公式の待ち時間」「メーカー推奨値」「修理費相場」を取得
   - 「しばらく」→「◯分」、「高額」→「◯万円」に変換
   - 検索結果にない数字は書かない（ハルシネーション厳禁）

2. 条件分岐を具体化する
   - 「場合による」は禁止
   - 「◯◯なら△△、□□なら××」と書く
   - 条件の境界値も検索で裏を取る

3. 判定基準を明示する
   - 「異常があれば」→「◯◯が△秒以上続いたら」
   - 五感（音・匂い・熱・見た目）で判定できる基準を書く

【AI Overviewとの差別化 - 必須】
1. 境界条件を必ず書く
   - 「◯分以内ならセーフ、△分以上ならアウト」
   - 境界値は検索で裏を取る（メーカー公式、専門家サイト優先）

2. 五感チェックを含める（該当するもののみ）
   - 聴覚：どんな音がアウトか（カタカタ、ブーン、ガリガリ等）
   - 嗅覚：どんな匂いがアウトか（焦げ臭い、ガス臭い等）
   - 視覚：何が見えたらアウトか（煙、液漏れ、変形等）
   - 触覚：触ってどうならアウトか（異常に熱い等）

3. 最悪ケース→安心の構造
   - 最悪ケース提示：「最悪◯◯になる（検索で修理費等を取得）」
   - 判定基準：「ただし◯◯がなければその心配は不要」
   - 安心：確率やデータがあれば記載（なければ書かない）

【各H3の構成順序】
{H2-1用またはH2-2用の構成順序}

【スマホで読みやすい文章構成】
- 1段落は2〜3文（50〜80文字程度）で改行する
- 長文は禁止。1文は40文字以内を目標
- 段落間は必ず空行を入れる（<p>タグで区切る）
- 箇条書きを積極的に活用

【必須フォーマット要素】
{装飾要素の説明}

【絶対禁止】
{禁止事項のリスト}
```

### 4.2 H2-1の役割（判定・診断系）

```
【このセクションの役割：H2-1「もうやってしまった人」のパニックに即答】
- 読者は「やってしまった直後」で焦っている
- 「セーフかアウトか」を即座に判定させる
- 境界条件（◯◯ならセーフ、△△ならアウト）を全H3に含める
- 五感チェック（音・匂い・見た目）で判定できる基準を示す
- 最悪ケース（修理代、故障リスク）を具体的に提示
- 「今すぐ何をすべきか」を1つだけ示す

【各H3の構成順序：H2-1用】
1. 断言の結論（「◯◯です。」「△△してください。」で即答）
2. 境界条件（セーフ/アウトの境目を数字や五感で示す）
3. 最悪ケース（やらかした場合の損失・修理代）
4. 安心材料（でも大丈夫な理由）
5. 今すぐの行動（何をすればいいか1つだけ）
※全部入れる必要はない。H3の内容に応じて取捨選択
```

### 4.3 H2-2の役割（手順・方法系）

```
【このセクションの役割：H2-2「これからやる人」に唯一の正解を示す】
- 読者は「正しいやり方」を知りたい（まだやっていない）
- 正しい手順を1つだけ示す（複数案を並べない）
- よくある間違い・やりがちなNG行動を必ず含める
- なぜその手順なのか（理由）を検索で裏を取って説明
- 公式・メーカー・業者の基準を引用する
- 「これだけやればOK」という安心感を与える

【各H3の構成順序：H2-2用】
1. 結論の手順（「◯◯してください」で即答）
2. なぜその手順か（メーカー公式、専門家の根拠）
3. よくある間違い（やりがちなNG行動）
4. 正しいやり方の詳細（ステップバイステップ）
5. 完了の確認方法（これでOKという基準）
※全部入れる必要はない。H3の内容に応じて取捨選択
```

## 5. 導入テンプレート

### 5.1 導入テンプレA：クライマックス法＋選択の自由

1. 結論を先出し
2. A/Bの2択に落とす
3. 読者の状況を当てに行く
4. この先で得られるメリットを約束

**例**
- 結論：原因はだいたい3つです。
- まずA「審査/制限」か、B「配信/データ不足」かを切り分けます。
- いまあなたが見ているのは、広告が"出ていない"のではなく、"出せない状態"の可能性が高い。
- この順番で確認すれば、無駄な操作が消えて、最短で前に進めます。

### 5.2 導入テンプレB：嫌いなこと回避＋認められたい欲

1. 最悪を短く言う
2. 読者を肯定する
3. だから先に守るポイントを提示する

**例**
- 事故が起きた瞬間、取り返しがつきません。
- でも、ここまで調べてるあなたは偉い。事故は"運"じゃなく"設計"で避けられます。
- 先に、最小の保険だけ作ります（3分で終わる）。
